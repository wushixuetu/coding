1 - 介绍

TestNG是一个测试框架，旨在简化广泛的测试需求，从单元测试（测试一个类别与其他类别隔离）到集成测试（测试整个系统由几个类，几个包甚至几个外部框架组成，如应用服务器）。

编写测试通常是一个三步过程：

  • 编写测试的业务逻辑并在代码中插入TestNG注释。
  • 在testng.xml文件或build.xml中添加有关测试的信息（例如，类名称，您希望运行的组等等）。
  • 运行TestNG。
你可以在欢迎页面找到一个简单的例子。

本文档中使用的概念如下所示：

  • 一个套件由一个XML文件表示。它可以包含一个或多个测试，并由<suite>标签定义。
  • 测试由<test>表示，并且可以包含一个或多个TestNG类。
  • TestNG类是包含至少一个TestNG注释的Java类。它由<class>标签表示，可以包含一个或多个测试方法。
  • 测试方法是在您的源代码中由@Test注释的Java方法。
TestNG测试可以通过@BeforeXXX和@AfterXXX注释来配置，它允许在某个点之前和之后执行一些Java逻辑，这些点是上面列出的任一项。

本手册的其余部分将解释以下内容：

  • 所有注释的清单和简短的解释。这将使您了解TestNG提供的各种功能，但是您可能需要查阅专门针对每个注释的部分以了解详细信息。
  • testng.xml文件的描述，其语法以及可以在其中指定的内容。
  • 各种功能的详细列表以及如何结合使用注释和testng.xml来使用它们。

以下是TestNG中可用的注释及其属性的简要概述。

@BeforeSuite：在此套件中的所有测试都运行之前，将运行带注释的方法。
@AfterSuite：在此套件中的所有测试都运行之后，将会运行带注释的方法。
@BeforeTest：在任何属于<test>标签内的类的测试方法运行之前，将会运行带注释的方法。
@AfterTest：在所有属于<test>标签内的类的测试方法都运行之后，将会运行带注释的方法。
@BeforeGroups：此配置方法将运行之前的组列表。这种方法保证在第一个属于这些组中的任何一个的测试方法被调用之前运行。
@AfterGroups：这个配置方法将在之后运行的组列表。这种方法保证在最后一个属于这些组中的最后一个测试方法被调用后马上运行。
@BeforeClass：注释的方法将在当前类中的第一个测试方法被调用之前运行。
@AfterClass：在当前类的所有测试方法运行之后，将会运行带注释的方法。
@BeforeMethod：注解的方法将在每个测试方法之前运行。
@AfterMethod：注解的方法将在每个测试方法之后运行。

dependsOnMethods 这个方法依赖的方法列表。
描述 这个方法的描述。
启用 是否启用此类/方法上的方法。

5.2 - 测试组

TestNG允许您执行复杂的测试方法分组。您不仅可以声明方法属于组，还可以指定包含其他组的组。然后可以调用TestNG并要求包含一组特定的组（或正则表达式），同时排除另一组。这给了你如何划分你的测试的最大灵活性，并且不需要你重新编译任何东西，如果你想连续运行两个不同的测试。

组在testng.xml文件中指定，可以在<test>或<suite>标记下找到。在<suite>标签中指定的组适用于下面的所有<test>标签。需要注意的是群体，这些标签是累计：如果指定组的“a” <套件>和“B” <测试>，那么这两个“A”和“B”将被包括在内。

例如，至少有两类测试是很常见的

  • 签入测试。在提交新代码之前，应该运行这些测试。他们通常应该快速，只要确保没有基本的功能被打破。
     
  • 功能测试。这些测试应涵盖软件的所有功能，并且每天至少运行一次，尽管理想情况下您希望持续运行它们。
通常，签入测试是功能测试的一个子集。TestNG允许你用测试组以非常直观的方式指定它。例如，你可以通过说你的整个测试类属于“functest”组，并且另外有几个方法属于组“checkintest”来构建你的测试：

有时候，你需要按照特定的顺序调用你的测试方法。这里有一些例子：

5.7.1 - 与注释的依赖关系

您可以使用在@Test注释中找到的属性dependsOnMethods或dependsOnGroups。

有两种依赖关系：
  • 硬依赖。您所依赖的所有方法必须已经运行并成功运行。如果您的依赖关系中至少发生一次故障，则不会在报告中调用并标记为SKIP。
  • 软依赖关系。即使其中一些失败，您仍将始终依靠您所依赖的方法运行。当你只是想确保你的测试方法按照一定的顺序运行，但是他们的成功并不取决于其他人的成功时，这很有用。通过在@Test注释中添加“alwaysRun = true”来获得软依赖关系。
这是一个硬依赖的例子：
查看源代码
打印？
@Test
public void serverStartedOk() {}
 
@Test(dependsOnMethods = { "serverStartedOk" })
public void method1() {}

在这个例子中，method1（）被声明为依赖于方法serverStartedOk（），这保证了serverStartedOk（）将总是被首先调用。

你也可以有依赖于整个组的方法：

查看源代码
打印？

@Test(groups = { "init" })

在这个例子中，method1（）被声明为依赖于任何匹配正则表达式“init。*”的组，它保证了方法serverStartedOk（） 和initEnvironment（）将始终在method1（）之前被调用。 

注意：如前所述，属于同一组的方法的调用顺序不能保证在测试运行中相同。

如果一个依赖的方法失败了，并且你有一个很难依赖的方法（alwaysRun = false，这是默认的），依赖它的方法不会被 标记为FAIL，而是被标记为SKIP。在最终报告中，跳过的方法将被报告（颜色既不是红色也不是绿色），这是非常重要的，因为跳过的方法不一定是失败的。

  • parallel =“instances”：TestNG将在同一个线程中运行同一个实例中的所有方法，但是两个不同实例上的两个方法将在不同的线程中运行。

此外，属性 线程数允许您指定应为此执行分配多少个线程。

注意：@Test属性timeOut以并行和非并行模式工作。

你也可以指定应该从不同的线程调用@Test方法。您可以使用属性threadPoolSize来实现此结果：

5.17 - 方法拦截器

一旦TestNG按照什么顺序计算测试方法将被调用，这些方法分为两组：
  • 方法按顺序运行。这些都是具有依赖或依赖的测试方法。这些方法将按特定顺序运行。

打印？

  List<IMethodInstance> result = new ArrayList<IMethodInstance>();
  for (IMethodInstance m : methods) {
    Test test = m.getMethod().getConstructorOrMethod().getAnnotation(Test.class);

      groups.add(group);
    }

      result.add(0, m);
    }
    else {
      result.add(m);
    }
  }
  return result;
}

5.22 - 改变套件（或）测试

有时，您可能只需要在运行时更改套件xml中的套件（或）测试标记，而无需更改套件文件的内容。

一个经典的例子是尝试和利用你现有的套件文件，并尝试使用它来模拟你的“被测应用程序”的负载测试。至少你会最终复制你的<test>标签的内容多次，并创建一个新的套件XML文件并使用。但是这似乎并没有很大的规模。

TestNG允许您在运行时通过监听器在套件xml文件中更改套件（或）测试标签。通过提供一个实现IAlterSuiteListener的侦听器来实现这一点。请参阅听众部分了解听众。

以下是一个示例，显示套件名称在运行时如何被更改：

查看源代码
打印？
public class AlterSuiteNameListener implements IAlterSuiteListener {
 
    @Override
    public void alter(List<XmlSuite> suites) {
        XmlSuite suite = suites.get(0);
        suite.setName(getClass().getSimpleName());
    }
}
此侦听器只能通过以下任一方式添加：
  • 通过suite xml文件中的<listeners>标签。
  • 通过服务加载器
这个监听器不能被添加到执行使用@Listeners注释。

6 - 测试结果

6.1 - 成功，失败和断言

如果测试完成而没有抛出任何异常，或者抛出了预期的异常（参见@Test注释中找到的expectedExceptions属性的文档），则认为测试成功。

您的测试方法通常由可引发异常或各种断言的调用（使用Java“assert”关键字）组成。一个“断言”失败将触发一个AssertionErrorException，这反过来将标记方法失败（如果没有看到断言错误，请记住在JVM上使用-ea）。

以下是一个示例测试方法：

查看源代码
打印？
@Test
public void verifyLastName() {
  assert "Beust".equals(m_lastName) : "Expected name Beust, for" + m_lastName;
}
TestNG还包括JUnit的Assert类，它允许您对复杂对象执行断言：
查看源代码
打印？

@Test

6.2 - 记录和结果

测试运行的结果在启动SuiteRunner时指定的目录中的名为index.html的文件中创建。该文件指向包含整个测试运行结果的各种其他HTML和文本文件。

使用Listener和Reporter与TestNG生成自己的报告非常简单：

  • 听众实现接口org.testng.ITestListener，并在测试开始，通过，失败等情况下实时通知...
  • 记者实现了接口org.testng.IReporter，并在TestNG运行所有套件时收到通知。IReporter实例接收描述整个测试运行的对象列表。
例如，如果要生成测试运行的PDF报告，则不需要实时通知测试运行，因此应该使用IReporter。如果你想写一个实时报告你的测试，比如一个带有进度条的图形用户界面或者文本记者在每次测试被调用时显示点（“。”），ITestListener就是你的最好的选择。
6.2.1 - 记录听众
这是一个显示“。”的监听器。对于每个通过的测试，每个失败都会有一个“F”，每个跳跃都会有一个“S”：
查看源代码
打印？
public class DotTestListener extends TestListenerAdapter {
  private int m_count = 0;
 
  @Override
  public void onTestFailure(ITestResult tr) {
    log("F");
  }
 
  @Override
  public void onTestSkipped(ITestResult tr) {
    log("S");
  }
 
  @Override
  public void onTestSuccess(ITestResult tr) {
    log(".");
  }
 
  private void log(String string) {
    System.out.print(string);
    if (++m_count % 40 == 0) {
      System.out.println("");
    }
  }
}
